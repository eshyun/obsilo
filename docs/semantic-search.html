Semantic Search

Obsilo's semantic search finds notes by meaning, not just keywords — using a local vector index that stays entirely on your machine.

How It Works

When you build the semantic index, Obsilo:

  1. Reads each note in your vault (Markdown + optionally PDFs, DOCX, PPTX, XLSX)
  2. Splits it into chunks (default: ~2000 characters, split at headings) with document title prefix
  3. Generates an embedding for each chunk via your configured embedding model
  4. Stores the vectors in a local SQLite database (sql.js WASM)
  5. Extracts your vault graph (Wikilinks, Tags, MOC-Properties) into the Knowledge DB
  6. Computes implicit connections between semantically similar notes

When you (or the agent) run a search, Obsilo uses a 4-stage retrieval pipeline: vector similarity, local cross-encoder reranking, graph expansion (1-3 hops via Wikilinks/MOC), and implicit connection discovery — finding notes by meaning, structure, and hidden connections.

Hybrid Search

Obsilo doesn't rely on semantic search alone. Every semantic_search call runs two searches in parallel:

  • Semantic search — cosine similarity on SQLite-stored vectors with adjacent chunk context
  • Keyword search — TF-IDF with stemming, stop-word filtering, and title boost

The results are combined using Reciprocal Rank Fusion (RRF) — notes appearing in both lists rise to the top. Then a local cross-encoder reranker (ms-marco-MiniLM, runs on-device via WASM) re-scores the top candidates for better precision.

Graph Expansion

After reranking, Obsilo expands results via your vault's knowledge graph. It follows Wikilinks and MOC-Properties (Themen, Konzepte, Personen etc.) for 1-3 hops to find structurally connected notes. Each expanded result shows the connection path: "via [[Artificial Intelligence]] (Themen)".

Implicit Connections

Obsilo discovers hidden connections between notes that are semantically similar but have no direct link. These are pre-computed in the background and shown both in search results and as suggestions in the sidebar. You can dismiss irrelevant suggestions — they won't appear again.

HyDE (Hypothetical Document Embeddings)

HyDE is an optional technique that improves recall for conceptual queries. When enabled, before embedding the user's query, Obsilo asks the LLM to write a short 2-3 sentence note excerpt that would answer the query. That hypothetical excerpt is then embedded instead of the raw query — giving the embedding model a much richer signal to match against.

Trade-off: HyDE costs one extra LLM call per search. It's off by default but can improve results noticeably for complex, conceptual questions. Enable it in Settings → Semantic Index → HyDE.

Setting Up the Index

Step 1: Enable Semantic Index

Go to Settings → Obsilo Agent → Semantic Index and toggle "Enable Semantic Index" on.

Step 2: Choose an Embedding Model

Go to Settings → Providers → Embeddings, click "Add Embedding Model", and configure one:

ModelProviderQualityNotes
text-embedding-3-smallOpenAIExcellentRecommended. Fast, cheap, 1536 dimensions. Requires OpenAI API key.
text-embedding-3-largeOpenAIBestHighest quality. 3072 dimensions. Requires OpenAI API key.
nomic-embed-textOllamaVery GoodFree, runs locally via Ollama. 768 dimensions. No API key needed.
mxbai-embed-largeOllamaExcellentFree, runs locally via Ollama. 1024 dimensions.
Changing the embedding model invalidates the index. If you switch models, rebuild the index from scratch — vectors from different models are not compatible.

Step 3: Build the Index

Click Build Index in Settings → Semantic Index. The first build takes a few minutes for a large vault. Progress is shown in the settings panel.

The index is stored as a SQLite database:

  • Knowledge DB: {vault}/.obsidian-agent/knowledge.db — vectors, graph, implicit connections (vault-specific)
  • Memory DB: {vault-parent}/.obsidian-agent/memory.db — sessions, recipes, episodes (shared across vaults)

Subsequent rebuilds are incremental — only files changed since the last build are re-indexed.

Auto-Indexing

Configure when the index is rebuilt automatically:

SettingDescription
startupRebuild when Obsidian opens (incremental — only changed files)
mode-switchRebuild when switching to a mode that uses semantic_search
neverOnly build manually (default)

There's also Auto-index on change — when enabled, files are re-indexed as you edit them. This keeps the index fully up to date but adds background API calls.

Using Semantic Search

The agent uses semantic_search automatically when you ask questions. You can also use it with filters:

Basic Search

"Find my notes about meeting planning and agenda creation"

Filtered Search

"Find notes about machine learning from the past month tagged with #research"

This maps to: semantic_search(query, folder, tags, since)

The RAG Pattern

When the agent runs semantic_search, it receives excerpts with the most relevant content already extracted. The agent is instructed to synthesize answers directly from these excerpts — not to call read_file on each result. This keeps the context window manageable and responses fast.

Tip If the agent seems unable to find notes it should know about, rebuild the index (Settings → Semantic Index → Build Index) to ensure recently added notes are included.
