# Obsilo Agent v2.3

**Versions 2.3.1 and 2.3.2** -- released April 2026.

This release cuts token costs by up to 90% for recurring tasks, overhauls the memory and self-learning system, and fixes critical issues with the ONNX reranker and implicit connections.

## Token cost reduction (EPIC-018)

A standard task like "search my notes on Kant and write a summary" previously consumed 634,000 input tokens across 8 LLM iterations, costing roughly $2 per execution. On GitHub Copilot's 168k context limit, the agent crashed outright. After this release, the same task uses ~60,000 tokens in 2-3 iterations at ~$0.15.

### Fast path execution (ADR-061)

When the agent has solved the same kind of task before, it can skip the iterative ReAct loop and execute a pre-planned tool sequence instead.

- Recipe matching detects known task patterns via keyword scoring with description fallback
- A single planner LLM call produces a concrete execution plan from the matched recipe
- The plan executes deterministically: search tools in parallel (stage 1), read tools in parallel (stage 2), then 1-2 normal loop iterations for presentation
- Guardrails: hard allowlists per stage (search-only, read-only), minimum 3 successful uses before qualifying, fallback to normal loop on invalid plans
- All tool calls still pass through the governance pipeline with approval checks and logging

### KV-cache-optimized prompt structure (ADR-062)

The system prompt is now ordered by stability to maximize KV-cache hits across iterations.

- Stable sections (mode definition, tool descriptions, rules, security boundary) sit at positions 1-8 and are cached across all iterations within a session
- Dynamic sections (skills, memory, recipes, vault context) sit at positions 9-16
- The timestamp moved from position 1 to position 16, fixing a bug where every API call invalidated the entire cache
- A recency anchor (current task list appended as last user message) compensates for the reduced primacy of skill instructions

### Context externalization (ADR-063)

Large tool results no longer accumulate in the conversation history.

- Results exceeding 2,000 characters are written to temporary files with compact references in the history
- Tool-specific reference formats: search results show top 5 matches, semantic search shows top 3 with scores, read_file shows headings and a preview
- Deterministic file paths (no timestamps) preserve KV-cache compatibility
- Append-only history: externalization happens at result creation time, never retroactively
- Automatic cleanup after task completion with orphan sweep on plugin startup

## Memory and learning improvements (ADR-058, 059, 060)

### Semantic recipe promotion (ADR-058)

Recipe promotion switched from sequence-based to intent-based matching. The previous system compared exact tool sequences, which never worked because LLMs choose tools non-deterministically. The new system compares user messages by cosine similarity using the same embedding model as the knowledge layer.

### Budget-aware long-term memory (ADR-059)

The long-term memory extractor now receives the 800-character budget as a hard constraint. Each entry carries a `[YYYY-MM]` recency tag. When new information would exceed the budget, the extractor removes or condenses the least relevant entries instead of letting the file grow past the visible window.

### Session summary reliability (ADR-060)

Fixed a bug where session summaries were written to the database but never read back. The MemoryRetriever now queries the database directly instead of looking for legacy `.md` files that were never created.

## ONNX reranker in Electron (FIX-07b)

The local reranker (Stage 4 of the knowledge pipeline) never worked because transformers.js detected Electron as Node.js and selected `onnxruntime-node`, whose native addon cannot load in the renderer process. Three changes fix this:

- esbuild plugin patches `IS_NODE_ENV` to `false` in the transformers.js backend selection, forcing the web/WASM branch
- WASM binary (`ort-wasm-simd-threaded.wasm`, 12 MB) is copied to the plugin directory and pre-loaded from disk to avoid CDN fetch
- RerankerService accepts `pluginAbsDir` for reliable WASM path resolution

A fail-once guard prevents repeated retries after an ONNX failure (FIX-07).

## Bug fixes

- **ImplicitConnections race condition (FIX-08):** The database could close during async computation, causing "Statement closed" errors. An `isOpen()` guard now prevents `computeAll()` from running on a closed database.
- **ChatLink YAML parse errors (FIX-11):** Malformed YAML frontmatter in linked notes is now logged concisely and the note is skipped instead of crashing.
- **Learned recipes force-enabled (FIX-10):** Recipe learning is now always active (the UI toggle was never wired up).
- **Rollup darwin-arm64:** Moved `@rollup/rollup-darwin-arm64` to optionalDependencies to fix CI builds on Linux runners.
- **Error handler lint:** Fixed `require-await` lint error in FastPath error handler callback.
